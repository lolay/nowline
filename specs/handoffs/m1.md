### 2. Langium Grammar (`@nowline/core`)

Define the `.langium` grammar file that produces a typed AST. The grammar must support all keywords and constructs from `specs/dsl.md`:

**Top-level keywords:**

- `config` ŌĆö optional preamble. Contains `scale` (display-only block with `name`/`label-every`/`label`), `calendar` (block with `days-per-week`/`days-per-month`/`days-per-quarter`/`days-per-year`, only valid with `calendar:custom` on roadmap), `style` (block with `bg`, `fg`, `text`, `border`, `icon`, `shadow`, `font`, `weight`, `italic`, `text-size`, `padding`, `spacing`, `header-height`, `corner-radius`, `bracket` sub-properties), and `default` (one-per-line: `default <entity> <properties>`).
- `roadmap [id] ["title"]` ŌĆö root declaration, one per file. Optional `author:"string"`, `start:YYYY-MM-DD`, `scale:<duration>`, `calendar:business|full|custom`.
- `include "path"` ŌĆö file inclusion with optional `config:merge|ignore|isolate` and `roadmap:merge|ignore|isolate` properties
- `person [id] ["title"]` ŌĆö individual contributor declaration. May be declared at roadmap top level or inline inside a `team` block. Declared at most once per merged scope. Bare `person <id>` inside a team is a membership reference.
- `team [id] ["title"]` ŌĆö team declaration with nesting (nested teams and inline person declarations or bare membership references)
- `anchor [id] ["title"] date:YYYY-MM-DD` ŌĆö named date on the timeline. `date:` is a required keyed property.
- `label [id] ["title"]` ŌĆö semantic tag / chip vocabulary. Optional `style:id`. Raw style properties not allowed on label lines. Undeclared labels used on items remain valid.
- `duration [id] ["title"] length:<duration literal>` ŌĆö named-alias declaration for a duration length. Universal Declaration Pattern (entity-style). `length:` is required. Universal properties (`description`, `link:`) also allowed. Must be declared before any entity referencing `duration:NAME`.
- `status [id] ["title"]` ŌĆö custom status value declaration. Universal Declaration Pattern (entity-style). Extends the built-in set (`planned`, `in-progress`, `done`, `at-risk`, `blocked`). Universal properties (`description`, `link:`) also allowed. Must be declared before any entity referencing `status:NAME`.
- `swimlane [id] ["title"]` ŌĆö grouping block with optional `owner:id`
- `item [id] ["title"]` ŌĆö work item inside swimlane/parallel/group
- `parallel [id] ["title"]` ŌĆö parallel execution block (children run in parallel, implicit join on dedent)
- `group [id] ["title"]` ŌĆö sequential item bundle (works inside swimlanes and parallel blocks)
- `milestone [id] ["title"]` ŌĆö achievement marker. Requires at least one of `date:` or `after:`. `after:` accepts a single id or a list of item/milestone/anchor ids.
- `footnote [id] ["title"] on:id` or `on:[id1, id2]` ŌĆö footnote annotation attached to one or more entities

**Universal properties** (valid on all entities): `labels`, `link`, `style` (sole visual property allowed on an entity), plus the `description` sub-directive (indented under the entity, not inline).

**Content vs. rendering separation.** Roadmap-section entities carry only semantic information (identity, ownership, sequencing, sizing, state, categorisation) plus an optional `style:id` reference. Raw style properties (`bg`, `fg`, `text`, `border`, `icon`, `shadow`, `font`, `weight`, `italic`, `text-size`, `padding`, `spacing`, `header-height`, `corner-radius`, `bracket`) may only appear in `style` blocks and `default <entity>` lines ŌĆö both in config. Using them on any roadmap-section entity (including `label`) is an error.

**Item properties**: `status`, `owner`, `after`, `before`, `duration`, `remaining`. `after` and `before` accept a single id or a bracketed list.

**Parallel/group properties**: `status`, `owner`, `after`, `before` (single or list). `duration` and `remaining` are NOT valid on parallel/group (computed from children).

**Required properties:**

- `item` requires `duration:`
- `anchor` requires `date:`
- `milestone` requires `date:` or `after:` (at least one)
- `footnote` requires `on:`

**Universal declaration pattern**: `keyword [id] ["title"] [key:value ...]`. Positional slots are only `id` (unquoted) and `title` (always double-quoted); at least one of the two is required. Every other property is keyed. All key-value pairs use `:`; `=` is not part of the v1 DSL. Every entity declaration (including `label`, `duration`, `status`) follows this pattern ŌĆö there are no positional named-alias carve-outs.

**Duration literals**: `\d+[dwmqy]` (days, weeks, months, quarters, years). Named aliases are declared in the roadmap section with `duration [id] ["title"] length:<value>` (e.g. `duration xs "Extra Small" length:1d`, `duration l "Large" length:3w`).

**Lists**: `labels:enterprise` or `labels:[enterprise, security]` ŌĆö single value without brackets, multiple with brackets.

**Comments**: `//` single-line, `/* */` multi-line.

See `specs/dsl.md` for the complete keyword table, property types, config block details, and examples.

### 3. Validation Rules

Implement the validation rules from `specs/dsl.md` ┬¦ "Value rules", "Include rules", and "Parallel and group rules". Groups to cover:

- **Structural** ŌĆö exactly one `roadmap` per file (included files governed by `roadmap:` mode); `config` must appear before `roadmap`; every entity has at least an `id` or a `title`; all ids are unique across the merged result (items, parallel, groups, anchors, persons, teams, milestones, footnotes share one namespace); at least one swimlane required.
- **Dates, durations, scales** ŌĆö `anchor` and dated `milestone` dates are valid ISO 8601; `duration:` values match `\d+[dwmqy]` or a name declared by a `duration` declaration in the roadmap section; `scale:` values must match the raw duration pattern (no name lookup); `duration` declaration ids are kebab-case and must not collide with the raw pattern or bare suffix letters (`d`, `w`, `m`, `q`, `y`); every `duration` declaration must specify `length:<duration literal>`; duplicate `duration` ids in a file are an error.
- **Calendar** ŌĆö `calendar:` on roadmap is one of `business`, `full`, `custom`; a `calendar` config block is only valid (and required) with `calendar:custom`, and must declare all four `days-per-week`, `days-per-month`, `days-per-quarter`, `days-per-year` as positive integers.
- **Required properties** ŌĆö `item.duration:`, `anchor.date:`, `milestone` requires `date:` OR `after:`, `footnote.on:`. Error messages name the entity by id/title and the missing property.
- **Status, labels, remaining** ŌĆö `status` is a built-in (`planned`, `in-progress`, `done`, `at-risk`, `blocked`) or a value declared by a `status` declaration in the roadmap section; `duration:`/`status:` references must resolve to declarations that appear earlier in the file (forward references are a validation error); `labels` values are kebab-case identifiers (undeclared labels are valid); `remaining` is a percentage (`0%`ŌĆō`100%`).
- **Styles and content/rendering separation** ŌĆö `style:` references on entities and labels must resolve to a declared style; style property values match their type/enum (see `dsl.md` rule 19); raw style properties may only appear in `style` blocks and `default <entity>` lines ŌĆö using them on any roadmap-section entity (including `label`) is an error.
- **Defaults** ŌĆö first positional argument after `default` must be a supported entity type (`item`, `label`, `swimlane`, `roadmap`, `parallel`, `group`, `milestone`, `footnote`, `anchor`); duplicate `default <entity>` for the same entity in one file is an error; `default <entity>` cannot set banned properties (see the "Banned on `default`" table in `dsl.md`).
- **Dependencies** ŌĆö `after:` and `before:` on any entity accept a single id or a bracketed list; every element must resolve to a declared item, milestone, anchor, parallel, or group id; no circular dependencies across the full graph.
- **Roadmap `start:`** ŌĆö ISO 8601; required whenever the file contains any `anchor` or any dated `milestone`; every anchor date and every dated milestone date must fall on or after `start:`; a malformed `start:` suppresses the two downstream rules until it is fixed.
- **Persons and teams** ŌĆö `person` may be declared at most once in the merged scope (a declaration is identified by a title or any keyed property); bare `person <id>` inside a team denotes membership and is valid with or without a separate declaration; a bare reference at top level with no declaration is a warning.
- **Footnote** ŌĆö `on:` references (single or list) must resolve to declared ids.
- **Include** ŌĆö paths resolve; no circular includes; `include` declarations appear before `config` and `roadmap`; duplicate includes of the same file are errors; `config:` and `roadmap:` modes must be `merge`/`ignore`/`isolate`; merge shadow warnings; isolate scope enforcement; on `roadmap:merge`/`roadmap:isolate` the parent and child must agree on `start:` (both absent or identical).
- **Parallel/group** ŌĆö `parallel` requires Ōēź2 children (single-child is a warning); `group` requires Ōēź1 child; `duration` and `remaining` not allowed on `parallel`/`group`; nesting rules per `dsl.md`.

Error messages should include file position and, where possible, suggest corrections (e.g., "did you mean 'auth-refactor'?").

### 4. TextMate Grammar

Create `grammars/nowline.tmLanguage.json` with token scopes as defined in `specs/ide.md`. This ships in m1 so syntax highlighting works immediately in VS Code and other TextMate-compatible editors ŌĆö no extension needed, just file type association.

### 5. Example Files

Create 3 example `.nowline` files:

- `minimal.nowline` ŌĆö single swimlane, a few items, demonstrates basic syntax
- `teams.nowline` ŌĆö person/team declarations, multiple swimlanes, anchors, milestones, footnotes
- `product.nowline` ŌĆö realistic product roadmap with config (styles), roadmap-section vocabulary (labels, durations, statuses), parallel/group blocks, owners, status, remaining, links, includes

### 6. Tests

- **Parser tests:** Verify the grammar parses all example files and produces the expected AST structure.
- **Validation tests:** Verify each validation rule from `specs/dsl.md` (┬¦ Value rules, ┬¦ Include rules, ┬¦ Parallel and group rules) triggers on invalid input and passes on valid input. Test error messages include position and suggestions.
- **Config tests:** Verify config merging, style resolution, defaults, and roadmap-section vocabulary (labels, durations, statuses) including entity-grammar parsing and forward-reference errors.
- **Include tests:** Verify include resolution, config/roadmap mode behavior (merge, ignore, isolate), circular include detection, and shadow warnings.
- **Parallel/group tests:** Verify nesting rules, implicit join semantics, and that `duration`/`remaining` are rejected on parallel/group.
- **Edge cases:** Empty file, file with only comments, file with only a roadmap declaration (no swimlanes ŌĆö should error), deeply nested structures.

Use Vitest for the test runner.

## Open Questions for m1

1. **Auto-generated ID slugification rules:** The spec confirms that when `id` is omitted but a `title` is present, the parser generates a kebab-case id by slugifying the title. The exact slugification rules (unicode handling, collision suffixes, length caps) are left to the implementer ŌĆö document the chosen rules in the grammar file and ensure they're stable across reruns.
2. **Property ordering:** Resolved ŌĆö the universal declaration pattern has positional `id` then `title` only; all keyed properties accept any order.
3. **Whitespace in lists:** `labels:[a, b]` vs `labels:[a,b]` ŌĆö both should work, but confirm the grammar handles this.
4. **Multi-line items:** Resolved ŌĆö line continuation via `\` at end of line. See design rule 8 in `dsl.md`.
5. **Include resolution timing:** Should includes be resolved during parsing (eager) or as a post-parse step? Eager is simpler but means file I/O during parse. Post-parse is cleaner for LSP.
6. **Legacy positional `duration`/`status` syntax:** Old positional forms (`duration NAME VALUE`, `status NAME`) are rejected in v1 ŌĆö only the entity form (`duration [id] ["title"] length:<value>`, `status [id] ["title"]`) is accepted. The parser must emit a clear migration-hinting error when it sees the old form, e.g. `"duration" requires "length:" ŌĆö did you mean "duration xs length:1d"?`.

These can be resolved during implementation. The answers should be documented in the grammar file and reflected in the example files.
