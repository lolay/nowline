Rendering Samples

pre-m2b · manual SVG

Five hand-drawn SVG roadmaps that stress the decisions in specs/rendering.md. These are for critique before the real layout engine lands — they are not generated from the DSL yet.

01

Minimal

The baseline: single swimlane, three items, hero now-line. Tests whether a "boring" roadmap still feels crisp.

DSL source

nowline v1

roadmap "Starter" start:2026-01-05 scale:1w

swimlane engineering "Engineering"
  item research "Research" duration:1w status:done
  item design   "Design"   duration:2w status:in-progress remaining:50%
  item build    "Build"    duration:3w status:planned
What this asks you to decide:
  • Timeline header is bottom-aligned with the title block. The title card (220×58) and the timeline header (shorter) sit in the same top-left region but the timeline's bottom edge matches the title card's bottom edge. The row above the timeline header stays empty (used only by the now-line pill floating up into the gap). This feels cleaner than top-aligned because the first item row then sits flush under both the title card's baseline and the timeline tick row.
  • Item bars sit inset inside each column (symmetric gutter). Each time-block (1w = 120px here) reserves its logical column, but the item's visual bar is inset 6px on each side for a 12px gap between back-to-back items. The gutter width matches the item's own internal text padding, so the chart feels evenly padded left-to-right. Sequentially abutting items (Research → Design → Build) show a clear visible gap instead of touching edge-to-edge. Dependencies and milestones still attach to logical column edges for date-accuracy; only the item's visual bar is shrunk.
  • Nowline attribution mark lives at the bottom-right of the drawing. When a footnote panel exists, the mark sits at the bottom-right of the panel; otherwise it sits at the bottom-right of the last swimlane (this sample: bottom-right of the Engineering lane). The mark is always the last thing drawn so it overlays its container and never extends below the overall drawing bounds. It's a small "now|ine" wordmark (scale 0.22), inset 12px from the container's bottom-right corner.
  • How prominent should the now-line be? Current: 2.25px red, "now" badge floating up into the row above the timeline header, no shadow.
  • Status dots sit in the upper-right corner of each item bar — out of the way of the title/subtitle text. Do we want a secondary glyph (check / clock / hourglass) for accessibility, or is color + position sufficient?
  • Progress stripe placement — bottom edge (current) vs. fill inside the bar. Bottom edge reads cleaner but is subtle.
  • Built-in status text (done/in-progress/planned) is not rendered; only the colored dot + duration/remaining. The dot's color carries the status name.
  • Single-swimlane panels don't alternate; is the lone PlantUML-style "tab" enough lane identity?
02

Platform 2026 — the full example

The canonical DSL sample: header logo, anchors, milestones, parallel + styled group, labels, footnotes, statuses, owners, links.

DSL source

nowline v1

config

style enterprise "Enterprise readiness"
  bg: blue
  text: white
style risky
  border: dashed
  fg: orange
style concurrent
  bracket: solid

default item shadow:subtle
default label style:subtle

roadmap platform-2026 "Platform 2026" author:"Acme Engineering" logo:"./brand/acme.svg" start:2026-01-06 scale:2w calendar:business

person sam "Sam Chen"
person jen "Jennifer Wu"

team platform "Platform Team"
team mobile "Mobile Team"

anchor kickoff     date:2026-01-06
anchor code-freeze "Code Freeze" date:2026-05-01
anchor ga-date     "GA Date"     date:2026-06-01

duration m  length:1w
duration l  length:2w
duration xl length:1m

label enterprise     style:enterprise
label security       style:enterprise
label low-confidence style:risky

status handoff "Handoff"  // custom status

swimlane platform owner:platform
  item auth-refactor "Auth refactor" duration:l after:kickoff \
    status:done owner:sam labels:enterprise \
    link:https://github.com/acme/auth/issues/123
  parallel after:auth-refactor
    group audit-track "Audit Track" labels:security
      parallel  // audit-log and audit-ui run concurrently inside the group
        item audit-log "Audit log v2" duration:l before:code-freeze \
          remaining:30% labels:[enterprise, security] \
          link:https://wiki.acme.com/audit
        item audit-ui "Audit UI" duration:m
    item sso "SSO plugins" duration:m labels:[enterprise, low-confidence]
  item platform-qa "Platform QA" duration:s status:handoff

swimlane mobile owner:mobile
  item offline "Offline mode" duration:l after:kickoff \
    owner:jen status:at-risk remaining:60%
  item push-v2 "Push notifications v2" duration:m owner:mobile

milestone beta      "Beta"      after:auth-refactor
milestone v1-ga     "v1 GA"     after:[auth-refactor, audit-log]
milestone ga-launch "GA launch" date:2026-06-01

footnote "Vendor dependency" on:audit-log
  description "Blocked until vendor contract is signed."
footnote capacity-risk "Team capacity risk" on:[mobile, platform]
  description "Mobile team is down to 2 engineers through Q2."
What this asks you to decide:
  • Anchors and milestones share a single marker row. Both live on the same horizontal band above the week-tick row, diamonds on the same baseline. Anchors are open diamonds, milestones are filled diamonds — two distinct glyphs in one row.
  • Collision rule: anchor-on-top. When two markers would occupy the same x-column (here: GA Date anchor + GA launch milestone both at Mar 16), the header row expands upward at that column: the anchor is bumped up into the slim band above, and the milestone stays in-row. Result: anchor diamond/label sits above, milestone diamond/label sits below, same column. This keeps labels readable and preserves the "anchor = input, milestone = output" mental model with anchor visually "earlier" than milestone.
  • Milestone verticals are prominent dashed ink. They read as "a hard gate that cuts the timeline": indigo-ink stroke (#1e1b4b) at 2px weight with a 6/4 dash pattern and round caps. Thicker and darker than the swimlane fills so they stay visible through a full-height traversal — clearly stronger than anchor verticals, which stay subtle dashed neutral (1px, 1/3 dash). A milestone whose predecessor overruns its line turns the dashed stroke deep red (#b91c1c) instead. Both milestones and anchors are drawn after the swimlane fills (like the now-line) so they overlay the lane background all the way down, emerging from the bottom tip of the marker-row diamond. Items still sit above the lines.
  • A date-driven milestone (GA launch, Mar 16) sits at its date. An after:-driven milestone sits at the rightmost predecessor's end — Beta at auth-refactor's end, v1 GA at audit-log's end. Earlier predecessors get a dark dotted arrow rightward to the line (visible: auth-refactor → v1 GA).
  • Anchor & milestone labels bias right of the diamond by default, flipping to the left only if the label would overflow the timeline area.
  • Symmetric column gutters on time-block items. Each item reserves its logical column width (1w = 70px here), but the rendered bar is inset 6px on each side for a 12px gap between back-to-back items — matching the item's own 12px internal text padding so the chart feels evenly padded left-to-right. Sequentially abutting items (Auth refactor → parallel region; Offline mode → Push v2) show a clear visible gap. Dependency arrows attach to the visual bar edges (so arrowheads land on the bar, not in the gutter), while milestone lines and parallel rails still snap to logical column edges for date-accuracy.
  • Nowline attribution mark lives at the bottom-right of the footnote panel. Since this sample has footnotes, the mark sits at the bottom-right of the footnote panel (not the last swimlane). It's the last thing drawn so it overlays panel content and never extends below the drawing bounds. Inset 12px from the panel's bottom-right corner.
  • Parallel inside a group: Audit Track contains a nested parallel so audit-log and audit-ui both start at the group's left edge and stack vertically (same start time, different durations). The group's inner dashed orange rails show the nested parallel boundary. The outer parallel (group + sso) has no style: applied, so it resolves to bracket:none and renders with no visual connector — its structure is conveyed entirely by the two tracks stacking at a shared start x.
  • Status dot moved to upper-right of every item bar; footnote superscript sits immediately to the left of the status dot (see "Audit log v2" — the red "1" and the blue status dot share the upper-right corner).
  • Id → title resolution. Whenever a reference has a defined title, render the title, not the id. owner: platform renders as owner: Platform Team. owner: sam renders as Sam Chen. Anchors with a string title (anchor code-freeze "Code Freeze") display "Code Freeze"; a title-less anchor (anchor kickoff) falls back to the id. Same rule for milestones, labels, groups, and swimlanes.
  • Link icon is always the external-link glyph (↗). Same shape for every link: value. The tile color encodes the provider (GitHub on auth-refactor, generic on audit-log — both ink in this sample, but the renderer also tints Linear indigo and Jira blue when those providers appear). One glyph, many tints — keeps the item bar legible and prevents a glyph-soup when providers multiply.
  • Status display = colored dot only for built-ins. done, in-progress, planned, at-risk, blocked have universally-understood colors, so their names add noise to every item bar. The dot alone is sufficient. Custom statuses (user-declared via status handoff "Handoff") still render their title as text next to the dot, since the user's custom color wouldn't be self-explanatory. Platform QA demonstrates the custom case — purple dot + "Handoff" label.
  • Label chiclets have padding to the progress strip. The item bar has two zones: a content zone (title/subtitle/label/link) and a bottom 4px progress strip. Label chiclets sit inside the content zone with a small gap above the strip, so the chiclet never sits flush against the progress bar.
  • The styled group (Audit Track) uses an orange tab label on top. Competes with the in-bar label chiclets if we ever stack lots of groups.
  • Owner display in small text below the title — sufficient, or do we need an avatar/initials bubble?

Platform 2026 — dark theme

Same layout, dark palette. Now-line stays red by spec; everything else inverts. Good sanity check for contrast on chiclets and status dots.

Theme difference only

// Same DSL as sample 02.
// Rendered with: nowline render roadmap.nowline --theme dark

// Light theme palette (for reference):
//   surface    = #f8fafc    panel      = #ffffff
//   ink        = #0f172a    ink-muted  = #64748b
//   grid       = #e2e8f0    now-line   = #e53e3e

// Dark theme palette:
//   surface    = #0b1220    panel      = #111a2e
//   ink        = #e2e8f0    ink-muted  = #94a3b8
//   grid       = #1f2a44    now-line   = #e53e3e

// Status dots & label bg colors keep the same hue,
// but item bar fills shift to darkened tints of the
// same hue (e.g. done = #064e3b instead of #ecfdf5).
What this asks you to decide:
  • The now-line red (#e53e3e) is the brand color — staying identical in dark mode makes sense, but is the contrast harsh? Alternative: slightly brightened #ef4444.
  • The orange "Audit Track" group tab picks up a saturated orange on a dark brown fill. Looks clean here but a pure-dark scheme might want a cooler orange variant.
  • Red overflow on the audit-log is in a deep #7f1d1d — still legible, still clearly "broken." Works?
  • Grid lines at #1f2a44 vs. the panel at #111a2e — subtle enough? They largely disappear in this mockup, which might be intentional or too far.
04

Dependencies, floating milestone & overflow

Cross-swimlane arrows, explicit [ ] brackets on the parallel block (with group-like padding), a floating milestone driven purely by after:, a dotted slack arrow from an earlier predecessor, and an item-level before: overflow rendered in red.

DSL source

roadmap "Dependency Demo" start:2026-01-05 scale:1w

anchor kickoff     date:2026-01-05
anchor code-freeze date:2026-03-02

swimlane backend
  item api "API v2" duration:2w after:kickoff
  parallel style:concurrent  // bracket:solid; no after: needed
    item search "Search service" duration:3w
    item audit  "Audit pipeline" duration:7w \
         before:code-freeze   // overflow → red
  item deploy "Deploy" duration:1w

swimlane frontend
  item sdk "SDK update" duration:2w after:api
  item ui  "New console UI" duration:3w \
       after:[sdk, search]

// No date: Beta floats to the rightmost predecessor's end.
milestone beta "Beta" after:[deploy, ui]
What this asks you to decide:
  • Floating milestone (no date, only after:). When a milestone has no explicit date:, it floats to the rightmost predecessor's visual end. Beta has after:[deploy, ui]; deploy is the later predecessor, so Beta's diamond + vertical line land right at deploy's visual end (x=1264). No red — by definition there's no overrun when the milestone lands wherever its predecessors land. This differs from Sample 2, where milestones with dates can be overrun and turn red.
  • Slack arrow from the earlier predecessor. ui finishes well before Beta. A dotted ink arrow from ui's visual right to the Beta line (at ui's row midline) communicates "ui finishes with slack; deploy is the binding predecessor." The arrow is dotted (not solid) specifically because it represents waiting time, not a data flow. We don't draw any arrow from deploy — its visual end coincides with Beta by construction.
  • Redundant after: on the parallel block is omitted. The parallel block flows immediately after api simply because it's the next construct in the swimlane and api is its spatial predecessor. Writing parallel after:api would be redundant (and slightly misleading — after: on a parallel would imply the whole block waits for all of api which is already the case). We drop it.
  • Parallel bracket = full [ ] with group-like padding. Both sides of the parallel render as proper brackets: [ on the left at logical x=520, ] on the right at logical x=1192. Each bracket has a short top and bottom serif. The brackets extend 12px above the topmost nested item (Search) and 12px below the bottommost (Audit pipeline) — the same vertical padding a styled group uses around its nested items. That framing makes fork + join visually explicit in a single glance, and mirrors the group's visual idiom so the two constructs feel kin.
  • code-freeze anchor stays at its date (Mar 02), independent of Beta. With Beta floated to the right of deploy, there's no longer a collision between code-freeze and a Beta column (compare Sample 2, where GA Date anchor + GA launch milestone at the same column triggered anchor-on-top stacking). The code-freeze anchor sits in-row; its dashed gray vertical runs through the audit bar's amber→red transition (the bar still renders in red past code-freeze, independently of the milestone).
  • Milestone line overlays the swimlane, prominently dashed. The Beta line uses the standard prominent treatment: indigo-ink (#1e1b4b) at 2px with a 6/4 dash pattern and round caps. It's drawn after the swimlane fills, so it appears just like the now-line. Deploy's right edge visually coincides with the Beta line (as expected, since Beta sits at deploy's end). Items render above lines, so deploy's solid right stroke wins over the dashed line at their intersection.
  • api → sdk arrow routes under the parallel bracket. The arrow exits api's visual right (x=514), drops below the bracket's bottom serif (y=288), steps right, and drops into sdk. This keeps the arrow outside the parallel block visually (it's a flow that bypasses the parallel, not one that participates in it).
  • Symmetric column gutters. 1w = 96px. Each item bar is inset 6px on each side, giving 12px between back-to-back items — matching the item's internal text padding. Dependency arrows attach to the visual bar edges. The red overflow zone on Audit pipeline keeps its logical extent so the visual overflow span tells the story.
  • Nowline attribution mark at bottom-right of the last swimlane (no footnotes in this sample → Frontend lane's bottom-right corner, inset 12px, drawn last so it overlays lane content and never extends below the drawing).
  • The item-level before:code-freeze overflow renders as a red trailing region with a "past code-freeze" label. That communicates the overflow without needing a separate red arrow up to the anchor. Keep this convention for item-level before: overrun?
  • Status dots in upper-right keep the left edge of the bar clean for title/subtitle. Confirmed here across both lanes.
05

Isolated include region

Native swimlanes above and below, a third-party "partner" roadmap rendered inside a dashed-border region. Same timeline. Dependency arrows cross the boundary.

DSL source

// primary.nowline
roadmap "Primary Release" start:2026-01-05 scale:1w

anchor kickoff date:2026-01-05

// Native swimlanes render FIRST, in declaration order.
swimlane core "Core"
  item core-api   "Core API"   duration:3w after:kickoff
  item migrations "Migrations" duration:2w after:core-api

swimlane integrations "Integrations"
  item bridge "Partner bridge" duration:2w \
       after:[migrations, partner.plugin-beta]

milestone ga "GA" after:bridge

// Isolated includes render LAST, below all native lanes, in include order.
include "partner.nowline" config:isolate roadmap:isolate

// partner.nowline (isolated — renders inside dashed region at bottom)
roadmap "Vendor plugin roadmap" start:2026-01-05

swimlane plugin "Plugin"
  item plugin-alpha "Plugin alpha" duration:4w after:kickoff
  item plugin-beta  "Plugin beta"  duration:2w after:plugin-alpha
What this asks you to decide:
  • Render order. Native swimlanes of the parent file render first (Core, then Integrations). The isolated include region follows at the bottom, in include order. Multiple includes would stack in declaration order beneath the native content.
  • Milestone and anchor lines cross the isolate region. The GA milestone's prominent dashed ink vertical (indigo-ink #1e1b4b, 2px, 6/4 dash) and the kickoff anchor's subtle dashed vertical (neutral gray, 1px) both run all the way from the marker row to the bottom of the chart — through the dashed isolate-region border and through its swimlanes. The isolate region is a visual container, not a timeline island: the primary roadmap's timeline still governs everything inside it.
  • Symmetric column gutters. 1w = 90px. Every item bar is inset 6px on each side, so Core API → Migrations show a clear 12px gap (matching the item's internal text padding) even though they are back-to-back, and Plugin alpha → Plugin beta are visibly separated inside the isolate region. Dependency arrows route to the visual bar edges so arrowheads land on the bar, not in the gutter.
  • Nowline attribution mark at bottom-right of the isolate region. No footnote panel, so the mark sits at the bottom-right of the last region — in this sample, that's the dashed isolate-include region (not the top Core lane). It renders on top of the region background, inset 12px from the bottom-right corner.
  • The dashed border uses a violet accent (#8b5cf6) to distinguish isolated content from native content. Is a distinct accent color the right treatment, or should the dashed border just use the neutral ink color?
  • The region label ("Partner · Vendor plugin roadmap") sits on the top edge of the dashed border with an external-link badge. Do we prefer a badge inside the region (like an icon watermark) instead of on the border?
  • Items inside the region render in the same style as native items but with a matching violet accent. That's a design choice — the spec doesn't dictate it. Worth codifying, or should we keep them visually identical to native items and let the border carry all the distinction?
  • Cross-region arrows (plugin-beta → bridge) cross the dashed border with no break. The long-haul kickoff → plugin-alpha arrow routes around the left edge of the isolate region to avoid piercing the Core API and Integrations bars. Keep that auto-routing behavior?
  • The GA milestone (after:bridge) renders as a solid black vertical — single predecessor, no overrun, no inbound connector arrows needed.
